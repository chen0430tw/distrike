# 把用户当傻逼：软件设计第零公理

> **简称：FIA（Forensic Inference Architecture，法证推断架构）**
>
> 核心命题：不信任任何单一声明，从现场物证交叉推断结论。假设用户/系统不会按预设流程配合，从现场证据多层确认，结论基于证据权重而非前提假设。

> 本文基于 Distrike 内测反馈、MOROZ/MSCM 算法设计经验，结合 HCI 领域相关研究，探讨"把用户当傻逼"作为软件设计方法论的理论基础与实践含义。

---

## 一、这个说法哪里来

"把用户当傻逼"这句话流传在中文互联网技术圈，和它接近的英文表达叫 **idiot-proof design**。但两者有微妙的差异——"傻逼"是一种设计假设，idiot-proof 是一种结果目标。前者是方法论，后者是工程规格。

羽中（Sunborn 联合创始人，《少女前线》制作人）提出的**弱保软理论**从游戏设计角度切入了同一个问题：软件/游戏对弱势用户的保护机制越完善，就越容易变成一种对用户智识的不信任，最终导致产品对所有人都变得乏味。弱保软的极端是"新手教程永不结束"，反弱保软的极端是"README 一行字，细节自己看源码"。两边都有人翻车。

真正的问题不是"用户笨不笨"，而是：**你的设计能否在用户做出你完全没预期的操作时仍然给出正确结果？**

---

## 二、主动用户悖论

1987 年，John Carroll 与 Mary Beth Rosson 在 *Interfacing Thought* 中提出了**主动用户悖论**（Paradox of the Active User）：

> 用户不读手册。他们上来就用。长期来看，提前学习会节省时间——但用户的实际行为不遵循长期最优，他们追求当下任务的完成。

这个悖论在三十八年后依然成立，甚至更严重，因为软件的可选择性暴增了。

Distrike 内测的第一个数据点完美地印证了这件事：

- README 第一行写了 `go install github.com/chen0430tw/distrike@latest`
- 也写了"Or grab the binary from Releases"
- 测试者打开 README，看到第一行有 `go`，以为需要先装 Go 环境，卡住

没有人读到第二行。不是因为他们笨，而是**主动用户会在第一个障碍处停下来尝试解决，而不是继续读文档**。这是人类的认知默认行为，不是个别现象。

Steve Krug 在 *Don't Make Me Think* 里把这个模式叫做"满意化"（satisficing）——用户不寻找最优解，他们接受第一个看起来可行的选项。第一个选项是"装 Go"，于是他们装 Go。

## 三、两种"把用户当傻逼"

这个命题有两种完全不同的实现方式，必须区分清楚。

### 3.1 蔑视式（错误的）

> "用户太蠢了，我们帮他们把选择权拿走。"

表现为：强制更新、隐藏高级选项、假设用户只会用默认值、用弹窗强制用户做你想让他做的事。这是 Linus Torvalds 批评的那种：

> "If you think your users are idiots, only idiots will use it."

这条路的终点是软件越来越像玩具，用户越来越不信任它。

### 3.2 防御式（正确的）

> "用户会做出我完全没预期的操作，我的系统必须在这种情况下仍然工作。"

不是蔑视用户的判断力，而是**对用户行为分布的诚实估计**。用户会：

- 把 Chrome 装在 `D:\软件\Chrome\` 而不是 `Program Files\Google\Chrome\`
- 把工具包解压到 `D:\Tools\` `D:\杂项\` `D:\Misc\` `D:\工具\`
- 移动安装目录后不更新注册表
- 不读进度条含义，看到程序无输出就以为卡死了

**这不是用户的错。这是设计者必须提前预算进去的操作空间。**

## 四、众口难调的本质

内测反馈里出现了一个典型的分歧：

- 测试者 A：扫出来的东西比火绒少（期待大而全）
- 测试者 B：某些项目只占几 KB，扫它有什么意义（期待精而准）

这不是用户在矛盾，这是**用户分布本身就是双峰的**。

KANO 模型把用户需求分成三类：基础需求（没有就不满意）、性能需求（越多越好）、魅力需求（有了惊喜）。扫描覆盖率对一部分用户是性能需求（越全越好），对另一部分是魅力需求（发现了预期外的大文件才有价值）。

解法不是折中，而是**分层设计**：

```
SAFE   → 自动列出，用户零决策成本
CAUTION → 列出 + 说明，用户确认
DANGER  → 仅列出，完全手动
[cosmetic] → 标注，让用户自己判断价值
```

这是一种把用户决策成本显式分配的架构。不是让软件替用户决定，也不是让用户对每条记录都做判断——而是**按风险层级预分配认知负担**。

## 五、MSCM：把文件系统当傻逼的算法

以上都是关于"如何对待人类用户"的讨论。但 Distrike 开发中遇到了一个更深的类比：**文件系统本身也是一个"不配合的用户"**。

传统方案（注册表查路径）假设系统会乖乖记录安装位置，然后从记录里取答案。这是"信任用户会按照预设流程操作"的设计——等效于"用户会把软件装在 Program Files 里"。

现实是：绿色软件、便携版、手动解压、用户自定义目录，注册表里什么都没有。

MSCM（Multi-Source Semantic Collapse Model）的思路是：**不问系统，直接从现场推断**。

对应到字典爆破的思维框架：

| 密码爆破 | 目录识别 |
|---|---|
| 候选字典（常见词/变体/leet） | 应用指纹库（Chrome/WeChat/QQ...） |
| Hash 比对（单一确认信号） | 多层门控（exe + 目录名 + 内部结构 + 猎物路径） |
| 命中 → 破解成功 | 多层同时命中 → 确认应用身份 |

具体地，MSCM 对每个候选目录计算五层信号：

1. **name_match**：目录名是否模糊匹配已知应用名（Chrome、微信、wechat...）
2. **marker_exe**：目录内是否存在特征可执行文件（chrome.exe、WeChat.exe...）
3. **marker_dir**：目录内是否存在特征子目录（User Data、nt_qq、tdata...）
4. **size_tier**：体积量级是否符合已安装应用预期
5. **prey_exists**：目标清理子目录是否在磁盘上存在

然后用模糊对称门控（Fuzzy Symmetric Gate）聚合：

```
quality = sqrt(跨层覆盖率 × 信号多样性)
gate    = 1 + 0.6 × tanh(3.0 × (quality - 0.5))
total   = linear_score × gate
```

**门控的核心作用**：单层信号（只有名字像）被压制到 0.13，不触发；三层以上同时命中被放大到 0.73 以上，确认。

这正是"把文件系统当傻逼"的具体实现——不相信任何单一信号，用多维交叉验证代替精确匹配，在"现场证据不完整"的情况下仍然给出可信的结论。

## 六、弱保软的边界在哪里

弱保软理论的核心矛盾：**保护越多，信任越少；信任越多，门槛越高**。

Distrike 的实践给出了一个具体的边界划分：

**应该保护的**：
- 用户不会读文档 → 扫描时打印进度，不能沉默
- 用户不知道哪些东西能删 → 每条猎物附带 Risk 等级和清理 Hint
- 用户会把软件装在奇怪的地方 → MSCM 自动识别，不要求用户手动配置路径

**不应该保护的**：
- 用户是否真的要删 → 让用户决定，软件只报告不执行（除非 `--yes`）
- 用户的文件是否有价值 → `FileStorage`/`FileRecv` 白名单保护，但媒体缓存由用户判断
- 用户选择安装到哪里 → 软件适应用户，不要求用户适应软件

这条线的本质是：**在感知层面保护用户（让用户能做正确的决定），在执行层面尊重用户（让用户的决定真正生效）**。

## 七、结论

"把用户当傻逼"不是侮辱，是一种**对用户行为分布的保守估计**，进而转化为设计约束。

它要求软件设计者提前回答：

1. 用户会在哪里做出我完全没预期的操作？（主动用户悖论）
2. 我的系统在这种情况下会崩溃、误报、还是优雅降级？
3. 我有没有把认知负担分配到正确的地方——软件承担发现，用户承担决策？

答案不是把所有决策权拿走，也不是把所有细节暴露给用户——而是**在正确的层次上划定软件与用户的责任边界**。

这是弱保软理论没有说清楚的部分：弱保的对象不是用户的智力，而是用户的**注意力预算**。用户的注意力是稀缺的，软件的职责是让用户把有限的注意力花在真正需要判断的地方，而不是花在"这个程序还在跑吗"上面。

## 八、白名单悖论：替用户后悔是一种傲慢

Distrike 的默认配置里有这样一段白名单：

```yaml
whitelist:
  - "*/Tencent Files/*/FileRecv"   # QQ 收到的文件
  - "*/Tencent Files/*/Msg3.0.db"  # QQ 消息数据库
  - "*/QQNT/*/nt_db"               # QQNT 消息数据库
  - "*/WeChat Files/*/FileStorage" # 微信收到的媒体
```

设计动机是善意的：防止用户一不小心把聊天记录删掉。但这里隐藏了一个关键的逻辑跳跃——**我们替用户预判了他"以后会后悔"**。

问题在于，"傻逼用户"有一个完全可以自洽的期望：**扫到越多越好**。

他不在乎以后。他现在 C 盘红了。他要的是数字变大。

Distrike 悄悄地在幕后帮他保留了 QQ 文件，他什么感知都没有——既没有感谢，也没有学到任何东西。下次 C 盘又红了，他还是会找工具扫，还是会好奇为什么扫出来的比别的工具少。

### 白名单是一种认知债务的延期支付

正确的问题不是"用户会不会后悔删掉 QQ 表情包"，而是：**用户有没有被告知这件事的代价？**

两个方案的本质区别：

| 方案 | 信息呈现 | 用户决策 | 后果 |
|---|---|---|---|
| 白名单（现状） | 隐藏，用户不知道存在 | 无决策机会 | 用户永远不理解为什么扫出来的少 |
| CAUTION 显示 | 列出 + 说明代价 | 用户自主决定 | 删了 → 表情包要重下 → 用户理解了代价 |

被表情包教训一次，用户就知道了"原来这个文件夹里存的是聊天图片，删了要重新下载"。**这个认知是白名单永远无法给他的。**

Carroll & Rosson 的主动用户悖论告诉我们：用户不读文档，但他们从错误里学习。白名单剥夺了用户犯错——进而学习——的机会。

### 真正需要保护的是不可恢复的

白名单该保护什么？看不可恢复性：

```
nt_db（消息数据库）     → 删了无法恢复  → 应该白名单 ✅
FileRecv（收到的文件）  → 删了无法恢复  → 应该白名单 ✅
nt_data/Pic（聊天图片） → QQ 从服务器重下 → 不应该白名单，CAUTION 显示 ✅
nt_data/Video（聊天视频）→ 同上          → CAUTION ✅
Cache（渲染缓存）       → 自动重建      → SAFE ✅
```

区分标准只有一条：**数据是否在别处存在副本**。聊天记录只在本地，白名单保护。聊天图片在腾讯服务器上有备份，用户可以决定要不要用磁盘空间换流量。

### 把这个逻辑推到极端

如果白名单的动机是"用户可能后悔"，那么逻辑上：

- 浏览器缓存也要白名单（下次打开页面会慢）
- pip 缓存也要白名单（下次 install 会重新下载）
- Steam 着色器缓存也要白名单（下次启动游戏会卡顿）

但没有人这样做，因为这些代价**用户可以接受，而且在接受代价的过程中理解了系统**。

QQ 聊天图片和 pip 缓存在语义上是同一类东西：**可以从网络重新获取的本地缓存**。对它们的保护标准应该一致——CAUTION 显示，用户决定，而不是替用户决定。

### 结论：白名单是给工具的，不是给用户的

真正的白名单语义应该是：**这个路径下面有无法恢复的用户数据，工具绝对不应该自动清理它**。这是对工具能力边界的约束，不是对用户信息的过滤。

把"用户可能后悔删"的东西也加进白名单，等于是在帮用户做判断——这是弱保软最典型的过度保护形态。它的代价是：用户对自己磁盘上的数据分布永远停留在无知状态，因为软件贴心地把"可能让他不舒服的信息"藏起来了。

**让用户看到，让用户决定，让用户从结果里学习。表情包重新下载一次，这个知识就永久地装进他脑子里了。**

## 九、简称速查：FIA

**FIA — Forensic Inference Architecture（法证推断架构）**

这套方法论的简称。以后说"FIA 一下"意指：

- **不信任声明**：不问注册表、不信文档、不假设用户按预设流程操作
- **从现场推断**：只看文件系统、目录结构、可执行文件等现场物证
- **多层交叉确认**：单一信号不足以定论，需要多层独立证据同时命中
- **证据权重决策**：用 MSCM 门控聚合模糊信号，而非要求精确匹配

FIA 在 Distrike 中的三个体现：

| 层次 | 传统方案 | FIA 方案 |
|---|---|---|
| 应用定位 | 查注册表 InstallPath | MSCM 五层信号推断 |
| 白名单设计 | 防止用户后悔 | 仅保护无服务器副本的数据 |
| 进度呈现 | 静默等待 | 每 2 秒报告现场计数 |
